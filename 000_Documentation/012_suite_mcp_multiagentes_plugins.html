Protocolo Model Context (MCP) + Azure AI

El Model Context Protocol define una arquitectura cliente-servidor para que un LLM, desde un cliente (VS Code, Cloud Desktop), se conecte de forma modular a múltiples servidores MCP (APIs/servicios locales o remotos).

Conceptos

  • Cliente MCP: IDEs / escritorios (VS Code, Cloud Desktop).
  • Servidor MCP: encapsula una fuente (local/API) y expone capacidades.
  • Ventajas: modularidad, extensibilidad, menor acoplamiento.
Analogía: un hub USB‑C que conecta varios servidores MCP a un único LLM.

MCP con toque Azure

  • Construye Agentes de Azure AI sobre Azure AI Search / Cosmos DB / APIs.
  • Encapsúlalos detrás de un servidor MCP para plug‑and‑play desde tu LLM.
  • Opcional: servidor único que expone varios agentes (Search/Writer/Web).

Diagrama

Cliente MCP (VS Code / Cloud Desktop) ├─ Servidor MCP A (Explorador de archivos local) ├─ Servidor MCP B (GitHub / Gmail APIs) └─ Servidor MCP C (Agente Azure AI → Azure Search / Cosmos / Web)

Sistemas multiagente con Azure AI + Semantic Kernel

Azure AI Agent Service crea agentes de dominio. Semantic Kernel actúa como orquestador para que colaboren mediante planes dinámicos.

Rol del orquestador

  • Decide qué agente llamar y cuándo.
  • Secuencia dinámica (A → B → C), condiciones y reintentos.
  • Planificador = “córtex prefrontal”.

Flujo ejemplo

Usuario → Planificador (intención) ├─ Agente Búsqueda (RAG) ├─ Agente Redacción (formatea) └─ Agente Guardado (persiste)

Snippet (Python, esqueleto)

from semantic_kernel import Kernel
from semantic_kernel.planning import Planner

kernel = Kernel()
# registrar wrappers a tus agentes de Azure como skills
# kernel.import_skill(...)

planner = Planner(kernel)
goal = "Busca X, redacta 150 palabras y guarda en /reports/x.md"
plan = planner.create_plan(goal=goal)
out = kernel.run(plan)
print(out)

Plugins en Semantic Kernel: Prompt • Integrados • Nativos

Los plugins son capacidades que el Kernel invoca. Tipos:

Prompt Templates

  • Definen estilo/comportamiento del LLM.
  • Útiles en agentes individuales.
  • No llaman APIs por sí solos.

Integrados (built‑in)

  • Preconstruidos para tareas comunes.
  • Sirven como “starter kit”.
  • Se extienden/ajustan según necesidad.

Nativos

  • En código (Python/C#): llaman APIs y ejecutan lógica.
  • Clave en multiagente con Azure Agents.
  • Permiten políticas de seguridad/observabilidad.

Patrón con Agentes de Azure

from semantic_kernel import Kernel
from semantic_kernel.functions import kernel_function

class AgentSearch:
    def __init__(self, client): self.client = client

    @kernel_function(name="search")
    async def search(self, query: str, top: int = 5) -> str:
        # Invoca agente "search" de Azure (pseudo)
        res = await self.client.agents["search"].run({"query": query, "top": top})
        return res["summary"]

kernel = Kernel()
kernel.import_plugin(AgentSearch(client), "agent_search")

Comparativa rápida

  • Prompt: estilo/consistencia de salida.
  • Integrados: productividad inmediata.
  • Nativos: integración con APIs/agentes y lógica de negocio.
